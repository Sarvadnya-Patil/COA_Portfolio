Activity 1: Performance Reflection
PLAYBOOK TASK:
"Why does your laptop feel slow even with a fast processor?" (real-world scenario)
Short reflection (150 words): Which unit affects performance the most and why?
Portfolio Submission:

A laptop with a fast GHz processor clock can still feel sluggish in real-world workloads due to systemic speed mismatches, primarily governed by the Memory and I/O units rather than the processing unit. While a modern CPU operates at high gigahertz frequencies—executing billions of clock cycles per second—accessing off-chip main memory (DRAM) suffers from a latency of roughly 50 to 100 nanoseconds. This latency forces the processing pipelines to wait hundreds of idle clock cycles for instruction code and variable packets to arrive (known as the 'Memory Wall'). Furthermore, I/O operations such as reading blocks from solid-state storage (SSDs) or fetching resources across background network threads introduce microsecond-to-millisecond delays. Consequently, overall system performance is heavily dominated by memory transfer bottlenecks and I/O interface latency, leaving the CPU pipelines stalled and making a fast laptop feel slow.

Activity 2: Harvard Architecture Mappings
PLAYBOOK TASK:
"Why microcontrollers use Harvard architecture?"
Syllabus Submission: Concept sketch / Scanned diagrams with 3 key differences. All metrics expanded below.
1. Simultaneous Concurrency (Instruction + Data Paths) ACTIVE (SPLIT CONCURRENCY)

Small microcontrollers need to complete execution in deterministic cycles. Harvard physically isolates code memory (Flash) from data memory (SRAM) with separate address and data bus lines. This enables the processor to fetch program instruction words and read/write RAM variables **in parallel on the same clock cycle**, whereas Von Neumann's shared bus structurally prevents this.

2. Predictable Timing & Hardware Optimization ACTIVE (DETERMINISTIC TIMING)

Embedded control loops require exact timing predictability. Having separate memory buses guarantees that memory variable reads/writes never stall instruction fetches, avoiding dynamic cycle jitters. Additionally, code bus widths can differ from data widths (e.g. 16-bit instruction Flash, 8-bit registers), saving silicon gates while maximizing throughput!

3. Structural Memory Protection & Code Security ACTIVE (CODE PROTECTION)

In microcontroller applications, safety is absolute. Since program instructions reside in physically distinct Flash ROM that is read-only during standard program loops, software variable data overflows in SRAM can never overflow and overwrite program code blocks. This structurally prevents buffer overflows, injection exploits, and sudden runtime crashes.

Fetch Cycle & Bus Interaction Concept (Solved):
When data and instructions share the same bus topology (Von Neumann), the bus becomes a single resource bottleneck. Fetching instructions and accessing variables cannot occur simultaneously. When Instruction 1 performs a memory load, the bus is occupied, forcing the next Instruction 2 fetch cycle to stall.
